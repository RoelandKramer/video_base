# FCDB Video & Data Analyser — Corner Module Plan

This document is the complete plan for the corner-kick module of the
FCDB Video & Data Analyser. It covers the visualisations to build, the
backend data flow, the integration of the thesis project's inference
library, the coach-feedback loop, and the retraining cycle.

The corner module is one section inside the broader Defending Corners
view of the analyser. Selecting a team in the analyser shows all of
that team's defending corners (clips) plus the seven analytical tiles
described below.

---

## 1. Goal of the module

For a chosen opponent, give coaching staff:

- A clear visual summary of how that team defends corners (where the
  ball is allowed to arrive, how their defenders organise, who marks
  whom, where the gaps are).
- One-click access to the original video clip of every aggregate, so
  any number can be sanity-checked against footage.
- A lightweight feedback path so coach corrections improve the model
  over time.

The thesis project supplies the trained models; the analyser is the
delivery surface.

Per match the backend calls
`CornerAnalyser.analyse_match(events, positions, team)` once, caches
the resulting list of corners, and on the team-summary screen
concatenates the per-match lists and calls `compute_aggregates(...)`
on the concatenated list.

## 3. What you receive from the thesis project

Four files. Drop them anywhere in the analyser repo (a `corner_models/`
subfolder is suggested).

| File                          | Purpose                                       |
|-------------------------------|-----------------------------------------------|
| `fcdb_corner_inference.py`    | The library code. One self-contained module.  |
| `defender_role_rf.joblib`     | Trained SQ1 (defender) classifier.            |
| `attacker_role_rf.joblib`     | Trained SQ2 (attacker) classifier.            |
| `feature_columns.json`        | Feature-column order used by both models.     |

These artefacts are regenerated by running
`python src/train_production_models.py` in the thesis project whenever
the labelling set is updated (see Section 9).

### 3.1 Prerequisites

Python ≥ 3.10. The following packages must be installed in the
analyser environment:

```
numpy>=1.24
pandas>=2.0
scipy>=1.10
joblib>=1.3
scikit-learn>=1.3
```

No other dependencies are required. The module does not touch the file
system except when the coach feedback path is used (Section 8).

### 3.2 Quick-start usage

```python
from fcdb_corner_inference import CornerAnalyser

analyser = CornerAnalyser(models_dir="corner_models")

# Analyse all defending corners of FC Den Bosch in a single match
corners = analyser.analyse_match(
    events_path    = "data/match_12345_events.json",
    positions_path = "data/match_12345_positions.json",
    defending_team = "FC Den Bosch",
)

# Pre-compute aggregates for the seven visualisations
aggregates = analyser.compute_aggregates(corners)
```

For a season-level dashboard, call `analyse_match` per match, append
each list, and pass the concatenated list to `compute_aggregates`.

## 5. The seven visualisations

### 5.1 Vis 1 — Ball-arrival zone heatmap

**Purpose.** Where this team allows the ball to arrive on defended
corners, and how often those zones lead to a shot.

**Background.** A 2-D attacking-half pitch sketch using the zone grid
defined in the thesis (the new graphic the user provided). The 12
central zones (GA1–PC3) plus the wider FRONT / EDGE / SHORT / OTHER
strips are outlined.

**Rendering rules.**

- Each zone is filled with a colour scaled by **shot rate** (the
  fraction of corners landing in that zone that produced a shot).
  Suggested scale: white → orange → red, 0 % → 100 %.
- Inside each zone, overlay two numbers:
  - top line: total corners landing there (`n`)
  - bottom line: shots / `n`
- Clicking a zone filters the per-corner browser below to only show
  clips whose ball arrival fell in that zone.

**Data source.** `aggregates["vis1_zones"]` and per-corner
`corner["delivery"]["zone"]`.

### 5.2 Vis 2 — Average defensive setup (slidable timeline)

**Purpose.** Where this team's defenders typically stand around a
corner, animatable from setup (t − 3 s) through the moment of delivery
(t = 0) to one second after (t + 1 s).

**Background.** 2-D attacking-half pitch sketch (same as Vis 1, but
without zone fill).

**Rendering rules.**

- Big dots = defenders. Each dot is labelled with the player's
  **jersey number**. The dot for the same defender keeps the same
  visual identity across all four time slices, so the user can see how
  a single defender drifts.
- Small dots = attackers (no jersey number, smaller radius). Default
  on; one toggle "Show attackers" deselects them entirely.
- Timeline slider snaps to four positions: `t-3000`, `t-1000`, `t0`,
  `t+1000` ms.
- Above the pitch, two tabs:
  - **With short defender** (corners where the opposing team uses a
    short-corner threat)
  - **Without short defender**
- Each tab pulls a different set of average positions because the
  defensive shape changes meaningfully when a short defender is
  occupied near the corner flag.

**Data source.**
`aggregates["vis2_average_positions"][with_short|without_short][defenders|attackers][t-3000|...|t+1000]`.

For per-defender continuity across slices: walk over the per-corner
data and group defenders by jersey within each `with_short` bucket;
the analyser is responsible for the per-jersey averaging because the
library only ships the cross-corner mean. (We can move that into the
library if the per-jersey average becomes a frequent need.)

### 5.3 Vis 3 — Per-corner role colouring (2-D pitch)

**Purpose.** For one specific corner, show who is marking man and who
is zonal.

**Background.** 2-D attacking-half pitch sketch.

**Rendering rules.**

- Attackers (the opposing team's players, since we are viewing
  defending corners): grey dots, small, no number.
- Man-markers (`role == "MAN"`): **red** dots, with jersey number.
- Zonal markers (`role == "ZONAL"`): **green** dots, with jersey
  number.
- Short defenders (`role == "SHORT"`): **blue** dots (only visible if
  the corner has a short defender).
- Counter defenders (`role == "COUNTER"`): **grey-outline** dots,
  shown near the halfway line.
- For each `MAN` defender, draw a thin red dashed line from that
  defender's dot to their assigned attacker's dot (the
  `marks_jersey` field of the library output).

**Data source.** `corner["defenders"][].role` and
`corner["attackers"][].role` plus positions.

### 5.4 Vis 4 — Per-corner role table

**Purpose.** The same information as Vis 3 in tabular form, for
analysts who prefer reading.

**Layout.**

```
┌──────────────────────────────────────────────────────────────┐
│ Defender name   │ #  │ Role    │ Marks (att jersey or zone) │
├──────────────────────────────────────────────────────────────┤
│ John Doe        │  5 │ MAN     │ 9                          │
│ ...             │    │         │                            │
└──────────────────────────────────────────────────────────────┘
```

**Data source.** `corner["defenders"][]`.

### 5.5 Vis 5 — Average role counts ("X Zonal · X Man · X Short · X Counter")

**Purpose.** A compact dashboard tile summarising the team's typical
defensive structure.

**Rendering rules.**

- Two rows, one per branch:
  - **If opponent has a player who comes short:**
    `4.1 Zonal · 3.2 Man · 1.0 Short · 0.7 Counter`
  - **If opponent does not have a player who comes short:**
    `5.0 Zonal · 2.5 Man · 0.0 Short · 1.0 Counter`
- Numbers are means across that subset; show one decimal.

**Data source.** `aggregates["vis5_role_counts"]["with_short"]` and
`["without_short"]`.

### 5.6 Vis 6 — Zones usually zonally marked

**Purpose.** Heatmap of which zones in front of goal this team
typically guards zonally.

**Background.** Same zone grid as Vis 1.

**Rendering rules.**

- Each zone is filled by the fraction of corners in which **at least
  one ZONAL defender** is standing in that zone at the kick instant
  (`position_at_kick`).
- Use the same colour scale as Vis 1 but with a green palette to
  distinguish it visually.
- Optionally label each zone with the percentage.

**Data source.** `aggregates["vis6_zonal_zones"]`.

### 5.7 Vis 7 — Attackers vs markers tile

**Purpose.** A single percentage tile: in what fraction of corners did
the attacking team have more genuinely attacking players in the box
than the defending team had man-markers? (Indicates the defence is
outnumbered; an exploitable inefficiency on the offensive side.)

**Rendering rules.**

- One big number: `41 %`, with a one-line caption: "of corners had
  more attackers in the box than man-markers".
- Add a sparkline showing the value per match if room.

**Data source.** `aggregates["vis7_attackers_exceed_markers_pct"]`.

### 5.8 Per-corner clip browser

Below the aggregates, list every defending corner in chronological
order with a thumbnail and the corner's clip embedded. Selecting one
corner expands Vis 3, Vis 4, and the small feedback option (Section
6) for that specific corner.

## 6. Coach feedback ("one small option")

The visualisation must include one small UI element per corner clip
that lets the coach correct role assignments. **It must remain small.**
A quiet dropdown beneath Vis 4 is the recommended placement; under no
circumstances should it dominate the per-corner browser.

### 6.1 UI flow

```
[ Vis 4 – role table for this corner ]
─────────────────────────────────────────
[?] Role assignment for this corner   ▾
    ◉ All roles are correct
    ○ There is a mistake

   ── when "There is a mistake" is chosen ──
   Player        #   Role           Marks
   John Doe      5   [MAN ▾]        [9  ▾]
   Jane Doe      6   [ZONAL ▾]      –
   ...
   [ Done ]                    [ Cancel ]
```

The editable list is the same as Vis 4 but every `Role` cell becomes
a dropdown and every `Marks` cell (active only for MAN rows) becomes
a dropdown of valid attacker jersey numbers. Coach edits whichever
rows are wrong, clicks **Done**.

Only synced corners are eligible for feedback (they are the only
corners shown on this dashboard anyway).

### 6.2 Path A — "All roles correct"

When the coach selects this option and confirms, the analyser calls:

```python
analyser.confirm_corner(
    corner_result      = corner_dict,
    labeling_sheet_path = "corner_models/labeling_sheet_v2.csv",
)
```

Effect: one row per player for that corner is appended to the
labelling sheet, with the auto-predicted role marked as the
confirmed `role`. If the corner already exists in the sheet, the old
rows are replaced.

### 6.3 Path B — "There is a mistake"

The expanded editable list is shown. When the coach clicks Done, the
analyser builds two dicts:

- `role_overrides = {jersey: new_role}` for every row where the role
  was changed.
- `marks_overrides = {jersey: attacker_jersey}` for every MAN row
  where the marked attacker was changed.

Then it calls:

```python
analyser.submit_role_corrections(
    corner_result      = corner_dict,
    role_overrides     = role_overrides,
    marks_overrides    = marks_overrides,
    labeling_sheet_path = "corner_models/labeling_sheet_v2.csv",
)
```

Same write semantics as Path A — full corner persisted, old rows
replaced if present, but using the corrected values.

### 6.4 What is persisted

Both paths write the same per-player row layout into the labelling
sheet:

| Column          | Notes                                              |
|-----------------|----------------------------------------------------|
| `corner_id`     | `"<match_id>:<start_time_ms>"`                     |
| `match_id`      | SciSports match ID                                 |
| `match_name`    | `"Home Team vs Away Team"`                         |
| `start_time_ms` | Kick timestamp                                     |
| `corner_side`   | "L" or "R"                                         |
| `player_team`   | "DEF" or "ATT"                                     |
| `jersey`        | Shirt number                                       |
| `x`, `y`        | Position at kick (attacking-right normalised)      |
| `auto_role`     | What the model predicted                           |
| `role`          | What the coach confirmed (same as auto, or edited) |
| `auto_marks`    | Model's Hungarian-assigned attacker (MAN only)     |
| `marks`         | Coach's confirmed marked attacker (MAN only)       |
| `confirmed_by`  | `"DV_ANALYSER"`                                    |
| `confirmed_at`  | UTC ISO timestamp                                  |

The thesis project's training pipeline consumes this CSV directly:
rows with `confirmed_by == "DV_ANALYSER"` are added to the training
set on the next model rebuild.

## 8. Coordinate system

All positions returned by the module are in the **attacking-right
normalised frame**:

- The attacked goal is always at `x ≈ +52.5`.
- The pitch centre is at `(0, 0)`.
- Positive `y` is the near-post side of the corner.

This means the analyser can use a single attacking-half pitch sketch
for every corner without mirroring based on which side the kick is
taken from.

## 9. Retraining loop

When enough coach corrections have accumulated in the labelling sheet:

1. Copy `labeling_sheet_v2.csv` back to the thesis project's
   `output/labels/` directory.
2. Run `python src/train_production_models.py` in the thesis project.
3. Replace the three `.joblib` / `.json` artefacts in the analyser's
   `corner_models/` folder with the newly produced ones.
4. Restart the analyser. The next `CornerAnalyser(...)` instance will
   pick up the retrained models.

No code changes are required in the analyser repo as part of the
retraining loop — the inference module is forward-compatible with any
model trained on the same feature set.

## 10. Performance characteristics

- One `analyse_match` call processes a 90-minute match in roughly 3–8
  seconds on a modern laptop, dominated by the time to load the
  position JSON.
- Memory usage during inference is bounded by one match's frames at a
  time (≈ 200 MB peak for a typical SciSports position file).
- The trained model artefacts together are ≈ 30 MB on disk.
- The two classifiers are warm-loaded once when `CornerAnalyser(...)`
  is instantiated. Re-instantiate only when the artefacts on disk
  change.
- For a team-summary screen across ~30 matches, expect ~2 minutes on
  first load (then cache the per-match results).

## 11. Error handling expectations

| Situation                                   | Library behaviour                                  |
|---------------------------------------------|----------------------------------------------------|
| Corner has `synced: false`                  | Silently skipped (not returned by `analyse_match`) |
| Corner with no defender tracking data       | Defender list will be empty; corner still returned |
| Attacker GK on attacking half               | Included as outfield attacker (rare end-of-match push) |
| `defending_team` does not appear in match   | `analyse_match` returns `[]`                       |
| Feedback call on a non-synced corner        | Raises `ValueError`                                |

`FileNotFoundError` at `CornerAnalyser(...)` means the `models_dir`
path is wrong.

## 12. Build phases (suggested)

A reasonable order of work for the analyser team:

1. **Backend wiring** — Add `fcdb_corner_inference.py` and artefacts.
   Build a thin caching layer that maps `(match_id, defending_team)`
   → list of corners. Add `(team, season)` → aggregates cache.
2. **Per-corner browser** — Implement the clip browser with Vis 3 and
   Vis 4. This is the smallest unit and the most useful for early
   coach feedback.
3. **Aggregate tiles** — Vis 1, 5, 6, 7 are essentially one chart
   each, fed by the aggregates dict.
4. **Vis 2 timeline** — The most complex view because it must
   group defenders by jersey across the time slices. Build after Vis
   3 to share the pitch-rendering component.
5. **Coach feedback panel** — Path A first (one-click confirm), then
   the editable list for Path B.
6. **Retraining handshake** — Documented procedure for shipping the
   updated CSV back to the thesis project.

The attacker-role visualisations are a separate later task (Vis 8+)
and use `corner["attackers"][].role` directly.

## 13. Open questions for later versions

Tracked in the thesis project's Plan chapter. They do **not** require
changes in the analyser repo:

1. The MAN/ZONAL boundary remains the largest source of classifier
   error; label expansion (including coach feedback) is expected to
   push accuracy above 95 %.
2. The attacker-role classifier currently treats short-corner
   attackers the same as direct-corner attackers; a later version may
   add a short-corner-specific branch.
3. Inter-rater reliability testing on the manual labels is pending,
   which will give an interpretable upper bound on classifier
   accuracy.
